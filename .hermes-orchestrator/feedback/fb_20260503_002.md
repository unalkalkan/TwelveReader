# PDF Parser Limitation — No Compressed Stream Support

## Observed behavior
Uploaded a 4-page PDF generated with fpdf2 (Python). The parser returned:
`Parse failed: pdf: no extractable text found`

The backend correctly identified the file as PDF (`%PDF-` header matches) and found content streams, but `extractTextFromStream` returned zero extractable text because:
1. fpdf2 outputs **FlateDecode**-compressed content streams
2. The custom parser in `internal/parser/pdf.go` only handles raw uncompressed streams
3. Modern PDF libraries universally compress their streams

## Root cause
`extractStreams()` extracts the raw bytes between `stream` and `endstream` markers but never decompresses them. FlateDecode (zlib/deflate) is the most common PDF stream filter and must be handled.

## Impact
- Any PDF produced by fpdf2, ReportLab, wkhtmltopdf, LibreOffice, or practically any modern tool fails to extract text
- PDF is the most common upload format users will try
- Only hand-crafted uncompressed PDFs work

## Requested fix
Add FlateDecode decompression to the PDF parser so that compressed content streams are decompressed before text extraction. The Go standard library has `compress/zlib` and `compress/flate` packages that handle this.

## Priority
High — blocks the primary upload format for real-world use.
